# Spec — MCP Mail macOS v1.1 (multi-comptes, répondre/transférer, CC/CCI, tests)

Date : 2026-06-23
Statut : validé (design approuvé par Maxence)

## Contexte

Le MCP « Mail macOS » v1.0 expose 8 outils pour piloter l'app Mail depuis Claude
Desktop (livré à Éric, élève de la formation). Trois manques pour un usage réel,
notamment multi-comptes :

1. **Envoi multi-comptes** : `creer_brouillon` / `envoyer_mail` partent toujours
   du compte par défaut de Mail. Impossible de choisir l'adresse expéditrice.
2. **Pas de réponse / transfert** : seule la création d'un mail vierge existe.
   Répondre à un fil (objet « Re: », destinataire, citation) n'est pas possible.
3. **Pas de CC / CCI** ni de **tests automatisés**.

Objectif : v1.1 additive (aucune régression), couvrant A (multi-comptes envoi),
B (répondre / transférer), C (CC/CCI), G (tests automatisés).

## Périmètre

### Nouveaux outils

- **`lister_comptes`** — liste les comptes connectés à Mail (nom + adresse(s)
  email). Lecture seule. Permet à Claude de savoir depuis quelle adresse écrire.
- **`repondre_mail`** — répond à un mail identifié par `id`. Reprend
  automatiquement le fil cité, le destinataire et l'objet « Re: ».
  Paramètres : `id` (requis), `corps` (requis, texte ajouté au-dessus de la
  citation), `repondre_a_tous` (booléen, défaut `false`), `cc?`, `cci?`,
  `expediteur?`, `envoyer` (booléen, défaut **`false`** = brouillon).
- **`transferer_mail`** — transfère un mail identifié par `id` vers un ou
  plusieurs `destinataire`. Paramètres : `id` (requis), `destinataire` (requis),
  `corps?` (mot d'accompagnement), `cc?`, `cci?`, `expediteur?`,
  `envoyer` (booléen, défaut **`false`** = brouillon).

### Outils enrichis (paramètres optionnels, rétrocompatibles)

- **`creer_brouillon`** — ajout de `cc?`, `cci?`, `expediteur?`.
- **`envoyer_mail`** — ajout de `cc?`, `cci?`, `expediteur?`.

Total : **10 outils** (8 + 2).

### Hors périmètre (YAGNI)

Pièces jointes, lister les dossiers, drapeaux/marquages, suppression définitive.

## Comportements clés

### Multi-comptes (`expediteur`)

- `expediteur` est une adresse email qui doit correspondre à un compte connecté.
- À l'envoi/brouillon, on positionne la propriété `sender` du message sortant sur
  le compte correspondant (forme « Nom <email> » attendue par Mail).
- Si `expediteur` ne correspond à aucun compte → erreur claire en français listant
  les comptes disponibles.
- Si `expediteur` absent → comportement v1.0 (compte par défaut de Mail).
- Pour `repondre_mail` / `transferer_mail`, en l'absence d'`expediteur`, Mail
  choisit automatiquement le compte adéquat (celui ayant reçu le mail d'origine).

### Sécurité (inchangée)

- `repondre_mail` et `transferer_mail` créent un **brouillon par défaut**
  (`envoyer: false`). L'envoi direct (`envoyer: true`) reste possible mais explicite,
  avec la confirmation de Claude Desktop.
- Aucune action irréversible (pas de suppression définitive).

### Citation / contenu (répondre & transférer)

- `repondre_mail` : le `corps` fourni est inséré **au-dessus** du texte cité du
  mail d'origine.
- `transferer_mail` : le `corps` optionnel est inséré au-dessus du contenu transféré.

## Architecture

Refactor ciblé pour rendre la génération de scripts testable sans Mail.

```
src/
  index.ts        # serveur MCP : déclare les 10 outils (validation zod)
  mail.ts         # orchestration : appelle un builder, exécute, parse
  scripts.ts      # NOUVEAU — fonctions PURES qui construisent les AppleScript
  applescript.ts  # runOsascript, asStr, séparateurs, parsing, erreurs FR
test/
  *.test.ts       # tests node:test (builders, parsing, échappement, validation)
```

- **`scripts.ts`** : un builder pur par opération
  (`buildListerScript`, `buildRechercherScript`, `buildLireScript`,
  `buildBrouillonScript`, `buildEnvoyerScript`, `buildRepondreScript`,
  `buildTransfererScript`, `buildMarquerScript`, `buildDeplacerScript`,
  `buildCorbeilleScript`, `buildListerComptesScript`). Chaque builder prend des
  options typées et renvoie une chaîne AppleScript. Aucune I/O.
- **`mail.ts`** : pour chaque opération, appelle le builder, `runOsascript`, puis
  parse. Contient aussi la logique de validation `expediteur` (à partir du retour
  de `lister_comptes`) et les helpers métier (`recipientsBlock`).
- **`applescript.ts`** : inchangé (helpers bas niveau réutilisés).

Frontière nette : `index.ts` (contrat MCP) → `mail.ts` (orchestration + validation)
→ `scripts.ts` (génération pure) → `applescript.ts` (exécution).

## AppleScript — points techniques

- **Expéditeur** : `set sender of newMsg to "Nom <email>"`. La valeur doit matcher
  une adresse d'un compte activé ; sinon Mail ignore ou erreur. On valide en amont.
- **CC / CCI** : `make new cc recipient` / `make new bcc recipient` au sein du
  `tell newMsg`, comme pour les `to recipient`.
- **Comptes** : `lister_comptes` parcourt `accounts`, renvoie `name`, et
  `email addresses` (liste) de chaque compte, sérialisés avec les séparateurs
  existants (FIELD_SEP / RECORD_SEP).
- **Répondre / Transférer** : commandes `reply` / `forward` de Mail. Risque connu :
  gestion capricieuse du contenu cité et de la fenêtre. Stratégie :
  `reply theMsg with opening window` (ou `without`), récupérer le message sortant,
  puis `set content of newMsg to (corps) & linefeed & (content of newMsg)` pour
  préfixer le texte de l'utilisateur. À **valider manuellement** sur un vrai mail.

## Gestion d'erreurs

- Réutilise `MailError` + `translateError` (autorisation refusée, Mail non lancé,
  mail introuvable, dossier introuvable).
- Ajout : `expediteur` invalide → message FR listant les comptes disponibles.

## Tests (node:test, sans dépendance)

Node 24 exécute le TypeScript nativement → `npm test` lance `node --test test/`.

Couverture (déterministe, sans Mail) :
- `asStr` : échappement guillemets, antislash, accents, sauts de ligne.
- `parseResumes` et parsing de `lire_mail` (séparateurs, champs manquants,
  corps contenant le séparateur).
- `recipientsBlock` : 1 destinataire, plusieurs (virgule/point-virgule), vide → erreur.
- **Builders de `scripts.ts`** : pour chaque outil et chaque option (cc, cci,
  expediteur, repondre_a_tous, envoyer), le script généré contient les bons
  fragments AppleScript (assertions sur sous-chaînes / structure).
- Validation `expediteur` : adresse connue → OK ; inconnue → erreur listant les comptes.

Non couvert par les tests unitaires (exécution réelle de Mail) → **vérification
manuelle** documentée pour `repondre_mail` et `transferer_mail`.

## Livraison

- `manifest.json` : passer à 10 outils, version `1.1.0`.
- `package.json` : version `1.1.0`, script `"test": "node --test test/"`.
- Rebuild du bundle : `npm run build` → `npm prune --omit=dev` →
  `npx @anthropic-ai/mcpb pack . mail-macos.mcpb` → `npm install`.
- Nouvelle **Release GitHub v1.1.0** avec le `.mcpb`. Le lien stable
  `releases/latest/download/mail-macos.mcpb` pointera automatiquement dessus.
- README : documenter les nouveaux outils et la notion de compte expéditeur.

## Vérification end-to-end

1. `npm test` vert (tous les tests unitaires passent).
2. `npm run build` sans erreur, `tools/list` renvoie 10 outils.
3. Sur Mail réel : `lister_comptes` cohérent ; `creer_brouillon` avec `expediteur`
   d'un compte secondaire → brouillon depuis la bonne adresse ; `repondre_mail`
   (brouillon) → réponse correcte dans Brouillons avec citation et bon destinataire ;
   `transferer_mail` (brouillon) → transfert correct.
4. Bundle installé dans Claude Desktop → les 10 outils visibles et fonctionnels.
